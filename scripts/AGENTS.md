# scripts 目录协作说明

本文件适用于 `scripts/` 目录及其子目录。

## 目录职责

- 本目录用于开发辅助、数据回填、诊断、验证和运维脚本
- 现有脚本以 `Node ESM (.mjs)` 为主，少量历史脚本使用 `Python` 或 `Shell`
- 新脚本优先复用已有业务模块，不要把核心业务逻辑长期复制到脚本里

## 编写原则

- 默认编写为一次性可执行、可重复运行、尽量幂等的脚本
- 涉及数据库、对象存储、文件删除、批量更新时，优先提供：
  - 明确日志
  - 错误处理
  - 安全边界
- 如果脚本会修改数据，优先支持“先检查、后执行”的流程；至少保证输出足够清晰，便于人工确认影响范围
- 不要在脚本里硬编码密钥、固定环境路径或生产专用配置

## 实现约定

- Node 脚本使用 `ESM import/export`
- 优先通过 `node:` 前缀导入 Node 内置模块
- 保持与仓库现有风格一致：
  - 双引号
  - 语句末尾分号
  - 变量名语义清晰
- 新增脚本文件名应直接表达用途，例如：
  - `backfill-*.mjs`
  - `test-*.mjs`
  - `audit-*.mjs`

## 与业务代码的关系

- 能调用 `server/`、`shared/` 中现有能力时，优先复用，不要重新实现同一套规范化逻辑
- 与前后端共用的数据结构、字段规范，优先从 `shared/` 引用
- 不要为了让脚本跑通而反向污染线上运行路径

## 验证建议

- 修改某个已有脚本时，优先运行该脚本本身验证
- 新增测试脚本时，文件名尽量与 `package.json` 中现有命名体系保持一致，便于后续挂到 npm scripts
- 不要提交临时输出、导出文件、缓存文件或 `__pycache__` 产物
